# Release v2.28.0 - Daemon Docs Guard + CI Passthrough Safety Fix

**Release Date:** 2026-03-25
**Type:** Minor Release

## Summary

Version 2.28.0 delivers a safety-critical fix for the CI passthrough bug that caused all safety handlers to permanently deactivate in non-CI environments, adds the new `daemon_docs_guard` advisory handler and `startup_cleanup` statusline handler, and fixes the upgrade script's version prefix handling.

## Highlights

- **Safety-critical: CI passthrough restricted to CI environments only** — Passthrough mode was incorrectly activating in containers and local dev machines on any transient daemon failure, permanently disabling all safety handlers. Now restricted to genuine CI environments (`CI`, `GITHUB_ACTIONS`, `GITLAB_CI`, `JENKINS_URL`, `TF_BUILD`). Also fixes a permanent passthrough lock bug.
- **`daemon_docs_guard` handler** — New PreToolUse advisory handler (priority 57) that warns when Claude Code reads from the daemon's internal `CLAUDE/` docs copy instead of the project's authoritative `CLAUDE/` directory.
- **Stale daemon file cleanup** — Age-based startup cleanup of dead-container runtime files and command-redirection output, with a `startup_cleanup` statusline indicator. Configurable via `stale_file_days` (default 7).
- **Upgrade script fix** — `scripts/upgrade.sh` now accepts version without `v` prefix (e.g. `2.28.0` and `v2.28.0` both work).

## Changes

### Added

- **`daemon_docs_guard` handler (PreToolUse, priority 57, advisory)**: Detects when Claude Code reads from the hooks-daemon's internal `CLAUDE/` docs directory instead of the project's own authoritative `CLAUDE/` directory. When the daemon is installed at `.claude/hooks-daemon/`, it brings its own `CLAUDE/` subdirectory which can collide with the project's `CLAUDE/` naming convention. The handler injects a corrective advisory message when a Read, Write, or Edit tool targets a path containing `hooks-daemon/CLAUDE/`, explaining which file was accessed and providing the correct project-level path. The operation is allowed (non-blocking). Includes 22 unit tests.

- **Stale daemon file cleanup with `startup_cleanup` statusline indicator**: Age-based cleanup on daemon startup removes runtime files (`daemon-*.*`) and `command-redirection/` output files older than a configurable threshold (default 7 days). Active daemons touch their runtime files hourly, so only files from dead containers age past the cutoff — enabling safe multi-container setups where multiple containers share one codebase. New `stale_file_days` config item added to `DaemonConfig` (range 1–365, default 7). New `startup_cleanup` Status handler (priority 28) shows a cleanup icon briefly after startup and a stale count for 30 seconds when files were removed.

### Fixed

- **CI passthrough activating in non-CI environments (safety-critical)**: Passthrough mode was incorrectly triggering in non-CI developer environments (containers, local machines) whenever the daemon had a transient failure, causing all safety handlers to permanently go inactive. Fixed by restricting passthrough activation to environments where a recognised CI variable is detected (`CI`, `GITHUB_ACTIONS`, `GITLAB_CI`, `JENKINS_URL`, `TF_BUILD`). Non-CI environments now receive a noisy error with restart instructions instead of silent passthrough. Also fixed a permanent passthrough lock bug where the `rm -f` cleanup flag after `start_daemon()` success was unreachable, preventing recovery after manual daemon restart.

- **Upgrade script version prefix normalisation**: `scripts/upgrade.sh` now accepts versions without the `v` prefix (`2.28.0` and `v2.28.0` are both valid inputs). Previously the internal `git rev-parse` validation only worked with the `v` prefix.

- **ShellCheck SC1091 source directive violations in `install_version.sh`**: Added proper `shellcheck source=` directives for all sourced modules in the install script, resolving pre-existing violations. Also reverted an incorrect Option A installer rename approach that made the cloned repo dirty; the handler-based Option B (`daemon_docs_guard`) is the correct solution.

## Upgrade Instructions

### For Existing Users

```bash
cd your-project
.claude/hooks-daemon/scripts/upgrade.sh
```

No configuration changes required. The `daemon_docs_guard` handler is automatically enabled after upgrade. Stale file cleanup runs automatically on startup.
