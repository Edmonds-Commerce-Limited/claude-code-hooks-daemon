# Release v2.27.0 - CI Environment Graceful Degradation

**Release Date:** 2026-03-23
**Type:** Minor Release

## Summary

Version 2.27.0 adds config-based graceful degradation when the hooks daemon is unavailable, solving a critical issue where Claude Code running in CI environments (GitHub Actions, GitLab CI, etc.) would enter investigation loops trying to restart the daemon.

## Highlights

- **CI environment graceful degradation**: New `ci_enabled` config option under `daemon:` in `hooks-daemon.yaml` controls behaviour when the daemon cannot start. Default: fail open with one-time warning then silent passthrough. `ci_enabled: true`: fail closed with STOP message for strict enforcement.

## Changes

### Added

#### CI Environment Graceful Degradation

When the hooks daemon is unavailable (no venv, no package installed), hook scripts now degrade gracefully instead of emitting confusing error messages that trigger agent investigation loops. The behaviour is controlled by a new `ci_enabled` option in `hooks-daemon.yaml`.

**Default behaviour (fail open):**
- First hook call: emits a one-time warning to stderr ("running in passthrough mode") and writes a `.hooks-passthrough` state file
- Subsequent calls: detects the state file and silently returns `{}` with no output
- If the daemon later becomes available, the state file is cleaned up automatically

**`ci_enabled: true` (fail closed):**
- Every hook call returns a hard deny (PreToolUse) or block (Stop/SubagentStop) with a STOP message
- Message clearly instructs the agent to stop and report that hooks daemon needs installing
- No state file is written — every call blocks consistently

**Implementation details:**
- All logic lives in `init.sh` (bash hook entry point layer) — zero changes to forwarder scripts or Python code
- Config parsing uses simple `grep` — no Python or yq dependency required in CI
- Function override pattern: `send_request_stdin()` is redefined to consume stdin and return `{}`
- Logic only activates when `ensure_daemon()` fails — zero impact on normal operation

**Files changed**: `init.sh`, `tests/unit/test_ci_passthrough.py`, `.claude/hooks-daemon.yaml.example`

### Fixed

#### `plan_number_helper` False Positive on Multi-Command Pipelines

The `plan_number_helper` handler was incorrectly blocking commands that had nothing to do with plan number discovery. The glob expansion regex (`echo\s+.*CLAUDE/Plan/[0-9*[]`) used `.*` between `echo\s+` and the plan directory path, allowing it to match greedily across command separators (`&&`, `||`, `;`, `|`).

This caused false positives in pipelines like:
```bash
DB_HOST=$(echo "$DATABASE_URL" | sed '...') && \
  mysql -e "$(cat /project/CLAUDE/Plan/00003-stats/report.sql)"
```

The regex matched `echo "$DATABASE_URL"` (unrelated) with the `CLAUDE/Plan/` path (in a `cat` command), incorrectly treating the whole pipeline as a plan number discovery attempt.

**Fix**: Replaced `.*` with `[^;&|]*` in the echo/printf glob patterns, preventing matches across command separator boundaries.

**Files changed**: `src/claude_code_hooks_daemon/handlers/pre_tool_use/plan_number_helper.py`, `tests/unit/handlers/pre_tool_use/test_plan_number_helper.py`

## Configuration

Add under `daemon:` section in `.claude/hooks-daemon.yaml`:

```yaml
daemon:
  # CI enforcement (v2.27.0+) - Controls behaviour when daemon cannot start
  # Default (false/omitted): fail open - warn once then silently passthrough
  # When true: fail closed - block all tool use and tell agent to stop
  # ci_enabled: false
```

## Upgrade Instructions

### For Existing Users

```bash
cd your-project
.claude/hooks-daemon/scripts/upgrade.sh
```

No configuration changes required. The default behaviour (fail open) requires no config and is backwards-compatible. Only set `ci_enabled: true` if you want strict enforcement.
