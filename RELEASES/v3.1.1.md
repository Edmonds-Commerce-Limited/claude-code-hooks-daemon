# Release v3.1.1 - Upgrade Path Fix for New Dependencies

**Release Date:** 2026-04-10
**Type:** Patch Release (Bug Fix)

## Summary

Version 3.1.1 fixes a critical bug where upgrading to v3.1.0 failed to install new dependencies (`mdformat`, `mdformat-gfm`), causing `ModuleNotFoundError` on daemon startup. The root cause was the idempotent upgrade path skipping venv recreation when Layer 1 had already checked out the target tag. Additionally, the post-install verification was silently passing despite the broken venv, masking the failure entirely.

This release also introduces venv version stamping (`.daemon-version` file) to reliably detect stale venvs on future upgrades, and resolves all shellcheck issues across install/upgrade scripts.

## Highlights

- **Venv recreation on idempotent upgrade path**: The upgrade script now detects stale venvs via a `.daemon-version` stamp file and recreates the venv when the version changes, ensuring new dependencies are correctly installed.

- **Post-install verification fails fast**: Post-install checks now call `validate_daemon_can_start()`, which verifies the daemon can actually import all modules. A broken venv is immediately caught and reported instead of silently passing.

- **Venv version stamping**: Both fresh install and upgrade scripts now write a `.daemon-version` file into the venv, enabling reliable stale-venv detection on future upgrades.

- **Zero shellcheck issues**: All shellcheck warnings across install/upgrade scripts resolved, including proper bash array handling for the Python interpreter flag in `venv.sh`.

## Fixed

- Venv not recreated on idempotent upgrade path — new dependencies (e.g. `mdformat`) now correctly installed when upgrading between versions
- Upgrade post-install verification now fails fast on broken venv instead of silently reporting success
- Venv version stamping (`.daemon-version`) added to detect stale venvs on upgrade
- `verify_venv()` in `venv.sh` now returns exit code 1 on import failure (was returning 0 with a warning)
- `upgrade_version.sh` post-install checks no longer swallowed by `|| true`
- All shellcheck issues resolved across `install_version.sh`, `upgrade_version.sh`, `venv.sh`, and manual test scripts
- Bash array handling for Python interpreter flag in `venv.sh` (SC2086 — word splitting on `$python_flag`)
- Dead code removed: unused `is_hooks_daemon_repo()` function in `detect_location.sh`

## Install / Upgrade

```bash
# New installation
git clone -b v3.1.1 https://github.com/Edmonds-Commerce-Limited/claude-code-hooks-daemon.git .claude/hooks-daemon
cd .claude/hooks-daemon && ./install.sh

# Upgrade from v3.1.0 or v3.0.x
cd .claude/hooks-daemon && git fetch && git checkout v3.1.1 && ./upgrade.sh
```
