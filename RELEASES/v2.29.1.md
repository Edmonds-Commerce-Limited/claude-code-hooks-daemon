# Release v2.29.1 - Stop Explanation Detection Fix

**Release Date:** 2026-03-31
**Type:** Patch Release

## Summary

Version 2.29.1 fixes a bug in `auto_continue_stop` where `STOPPING BECAUSE:` was only recognised when it appeared at the very first character of the assistant's last message. Assistants naturally write a summary of completed work before stating the stop reason, so the prefix almost always appeared on a later line — causing the handler to deny valid stops and force unnecessary retry loops.

## Highlights

- **`auto_continue_stop` multi-line fix** — `STOPPING BECAUSE:` is now recognised on any line of the assistant's last message, not just at position zero.

## Changes

### Fixed

- **`auto_continue_stop` stop explanation recognised on any line**: `_has_stop_explanation()` previously used `startswith()` on the entire assistant message, so `STOPPING BECAUSE:` was only recognised when it appeared at the very first character. Assistants naturally summarise completed work before stating the stop reason, placing the prefix on a later line — causing the handler to deny valid stops and force unnecessary retry loops. Fixed by scanning each line individually so the prefix is recognised wherever it appears in the message. Three regression tests added covering: prefix after a multi-line summary, prefix as the final line, and prefix with leading whitespace on a non-first line.
