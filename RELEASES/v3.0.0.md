# Release v3.0.0 - Command Redirection Removed

**Release Date:** 2026-04-07
**Type:** Major Release (Breaking Changes)

## Summary

Version 3.0.0 removes the `command_redirection` feature entirely. This was an internal mechanism on the `pipe_blocker`, `npm_command`, and `gh_issue_comments` handlers that, when enabled, stripped the blocked portion of a command, executed the remainder, and saved the output to a file so Claude could see the result without re-running the command.

In practice the feature caused unpredictable behaviour, could not handle output redirection correctly, and added significant complexity and corruption risk for marginal benefit. It has been disabled by default since v2.31.0. v3.0.0 removes it entirely along with the shared `core/command_redirection.py` module, its tests, config options, documentation, and QA exclusions.

## Highlights

- **`command_redirection` option removed** from `pipe_blocker`, `npm_command`, and `gh_issue_comments` handlers — unrecognised in config and silently rejected by schema validation.
- **`core/command_redirection.py` module deleted** — `execute_and_save`, `launch_and_save`, and `format_redirection_context` are no longer importable.
- **Affected handlers now simply deny** the blocked command with educational guidance, leaving Claude to construct the corrected command itself.
- **Daemon cleanup simplified** — `cleanup_stale_command_redirection_files()` removed from `daemon/paths.py`. Daemon startup only cleans stale runtime files now.
- **17 files changed, 1,829 lines deleted** — significant net reduction in surface area and complexity.

## Breaking Changes

**See the full upgrade guide:** [CLAUDE/UPGRADES/v3/v2.32-to-v3.0/README.md](../CLAUDE/UPGRADES/v3/v2.32-to-v3.0/README.md)

### `command_redirection` option removed

**Affected handlers**: `pipe_blocker`, `npm_command`, `gh_issue_comments`

**What broke**: Any project that has `options: command_redirection: true` (or `false`) under one of these handlers in `.claude/hooks-daemon.yaml` will fail config schema validation on daemon startup.

**Migration**: Remove the `command_redirection` line from each affected handler's config. The handlers retain their core blocking behaviour — they just no longer attempt to execute a corrected version of the command on Claude's behalf.

### `core/command_redirection.py` deleted

**Affected**: Any project handler or third-party code that imports from `claude_code_hooks_daemon.core.command_redirection`.

**Migration**: There is no replacement. If your project handlers depend on `execute_and_save`, `launch_and_save`, or `format_redirection_context`, you will need to inline the small amount of subprocess logic you need or rethink the approach.

### `cleanup_stale_command_redirection_files()` removed

**Affected**: Any code that imports `cleanup_stale_command_redirection_files` from `claude_code_hooks_daemon.daemon.paths`.

**Migration**: Remove the import. Daemon startup cleanup now only handles `cleanup_stale_daemon_files()`, which is unchanged.

## Changes

### Removed

- **BREAKING: `command_redirection` feature removed entirely.** The `command_redirection` option on `pipe_blocker`, `npm_command`, and `gh_issue_comments` handlers has been removed. The shared `core/command_redirection.py` module has been deleted along with all associated tests, config options, documentation, and QA exclusions.

### Changed

- **BREAKING: Affected handlers now simply deny with educational guidance.** `pipe_blocker`, `npm_command`, and `gh_issue_comments` no longer attempt to execute a corrected version of the blocked command. They return a `deny` decision with a clear explanation, leaving Claude to construct the corrected command itself. This is the same behaviour these handlers had with `command_redirection: false` (the default since v2.31.0).

- **Daemon stale-file cleanup no longer cleans command redirection files.** The `cleanup_stale_command_redirection_files()` function has been removed from `daemon/paths.py`. Daemon startup cleanup now only removes stale daemon runtime files (sockets, PID files, logs).

## Upgrade Instructions

**Required**: Read the full upgrade guide before upgrading: [CLAUDE/UPGRADES/v3/v2.32-to-v3.0/README.md](../CLAUDE/UPGRADES/v3/v2.32-to-v3.0/README.md)

### Quick Upgrade (No `command_redirection` in your config)

If you have never explicitly set `command_redirection` in your `.claude/hooks-daemon.yaml`, the upgrade is a standard daemon update:

```bash
cd .claude/hooks-daemon
git fetch --tags
git checkout v3.0.0
untracked/venv/bin/pip install -e .
untracked/venv/bin/python -m claude_code_hooks_daemon.daemon.cli restart
```

### Upgrade with config changes (You have `command_redirection` set)

1. Edit `.claude/hooks-daemon.yaml` and remove any `command_redirection:` lines under `pipe_blocker`, `npm_command`, or `gh_issue_comments`. If the `options:` block is now empty, remove the empty `options:` line as well.
2. Apply the daemon upgrade:
   ```bash
   cd .claude/hooks-daemon
   git fetch --tags
   git checkout v3.0.0
   untracked/venv/bin/pip install -e .
   untracked/venv/bin/python -m claude_code_hooks_daemon.daemon.cli restart
   ```
3. Verify daemon status: `untracked/venv/bin/python -m claude_code_hooks_daemon.daemon.cli status` — should show `RUNNING`.
