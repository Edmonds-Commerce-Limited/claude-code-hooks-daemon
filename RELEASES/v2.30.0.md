# Release v2.30.0 - Handler Self-Documentation and CLAUDE.md Injection

**Release Date:** 2026-04-02
**Type:** Minor Release

## Summary

Version 2.30.0 introduces mandatory handler self-documentation via a new `get_claude_md()` abstract method on the `Handler` base class, and a new `ClaudeMdInjector` core component that automatically writes all active handler guidance into the project `CLAUDE.md` on daemon restart. This means every Claude Code session now receives up-to-date, handler-generated context about what the daemon is doing and why — without any manual documentation effort. Also includes a targeted fix to a remaining stale transcript race condition in the stop hook explanation detection.

**⚠️ Custom handler authors:** `get_claude_md()` is now an abstract method. Any custom project handler that does not implement it will fail to load after this upgrade. Adding `return None` is sufficient — see Upgrade Instructions below.

## Highlights

- **Mandatory handler self-documentation** — `get_claude_md()` is now an abstract method on `Handler`; all 77+ handlers have been updated with explicit stubs or real guidance strings.
- **Automatic CLAUDE.md injection** — `ClaudeMdInjector` writes a `<hooksdaemon>` section into the project `CLAUDE.md` on every daemon restart, keeping in-session guidance in sync with active handlers.
- **`pipe_blocker` and `auto_continue_stop` publish guidance** — these two high-impact handlers now explain their behaviour and expected patterns directly in CLAUDE.md.
- **Stop hook race condition fixed** — a remaining stale transcript race condition (not fully addressed by v2.29.2's retry loop) is now resolved by requiring a confirmed fresh assistant message before accepting a stop explanation.
- **⚠️ Soft breaking change for custom handler authors** — `get_claude_md()` is now abstract; any custom project handler that does not implement it will fail to load. Adding `return None` is sufficient.
- **Release process audit step added** — Step 7.6 in the release documentation mandates a CLAUDE.md guidance audit before each release.

## Changes

### Added

- **`get_claude_md()` abstract method on `Handler` base class**: Every handler must now explicitly implement `get_claude_md()`, returning either `None` or a guidance string describing its behaviour. This makes handler self-documentation mandatory and opt-out rather than opt-in, preventing silent omissions. All 77+ existing handlers have been updated with explicit `return None` stubs or real guidance strings.

- **`ClaudeMdInjector`**: New core component (`src/claude_code_hooks_daemon/core/claude_md_injector.py`) that collects `get_claude_md()` output from all active handlers on daemon restart and writes a `<hooksdaemon>...</hooksdaemon>` section into the project `CLAUDE.md`. The section is auto-replaced on each restart and never duplicates. Invoked from `DaemonController.initialise()` after all handlers are loaded. Includes 13 unit tests.

- **`pipe_blocker` publishes guidance via `get_claude_md()`**: Returns a guidance block explaining what commands are blocked and showing the correct redirect-to-temp-file pattern as an alternative.

- **`auto_continue_stop` publishes guidance via `get_claude_md()`**: Returns a guidance block explaining the `STOPPING BECAUSE:` prefix requirement so agents know how to signal intentional stops without triggering retry loops.

- **Step 7.6 CLAUDE.md guidance audit added to release process**: The release documentation now includes a mandatory sub-agent analysis step that verifies all impactful handlers have accurate `get_claude_md()` content before each release.

### Fixed

- **Stale transcript race condition in `_has_stop_explanation`**: A second Stop event could fire before the new assistant turn was written to the transcript. `get_last_assistant_message()` then returned the previous turn's message (which contained `STOPPING BECAUSE:`), incorrectly allowing the new stop without a fresh explanation. Fixed by also retrying when the last message in the transcript is not from the assistant, only accepting an explanation when the last message is confirmed to be an assistant message with text blocks belonging to the current turn. If retries are exhausted without a fresh message, the method returns `False`.

## Breaking Changes

This release contains a breaking change for users with custom project-level handlers. **Read the upgrade guide before upgrading.**

**Breaking Change:** `get_claude_md()` is now an abstract method on the `Handler` base class. Any custom project handler that does not implement it will fail to load.

**Upgrade Guide:** [v2.29 → v2.30](../CLAUDE/UPGRADES/v2/v2.29-to-v2.30/)

**Quick fix:** Add `def get_claude_md(self) -> str | None: return None` to each custom handler class.

**Detect affected handlers:**
```bash
cd .claude/hooks-daemon
untracked/venv/bin/python -m claude_code_hooks_daemon.daemon.cli validate-project-handlers
```

## Upgrade Instructions

### For Existing Users

```bash
.claude/hooks-daemon/scripts/upgrade.sh
```
