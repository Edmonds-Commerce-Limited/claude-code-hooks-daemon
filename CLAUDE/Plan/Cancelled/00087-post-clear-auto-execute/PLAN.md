# Plan 00087: Post-Clear Auto-Execute

**Status**: Cancelled (2026-03-11)
**Created**: 2026-03-11
**Owner**: Claude
**Priority**: Medium
**Recommended Executor**: Sonnet
**Execution Strategy**: Single-Threaded

## Overview

When using `/clear execute plan 85` in Claude Code, the session clears and the new context contains the text "execute plan 85", but the agent just sits idle waiting for further input. The user has to manually type "go" or similar to kick things off.

This plan explored whether we can use a hook (likely SessionStart or UserPromptSubmit) to detect that `/clear` was followed by instructional text, and automatically inject a "go" signal so the agent acts on it without manual prompting.

## Outcome: Cancelled — Fundamental Client-Side Limitation

The investigation confirmed that hooks **cannot solve this problem**. The limitation is in Claude Code's client-side `/clear` command behaviour, not in the hook system.

## Investigation Findings

### Finding 1: Post-clear text IS visible in hook events

- **SessionStart** fires with `source: "clear"` (can be matched with `matcher: "clear"`)
- **UserPromptSubmit** fires on the first user prompt after clear, with `session_id` change detectable
- Both events support `additionalContext` injection into Claude's context

### Finding 2: The `local-command-caveat` blocks action

When `/clear <text>` runs, Claude Code wraps the output in:

```xml
<local-command-caveat>
Caveat: The messages below were generated by the user while running local commands.
DO NOT respond to these messages or otherwise consider them in your response
unless the user explicitly asks you to.
</local-command-caveat>
```

The command-args text (`<command-args>let's continue with plan 87</command-args>`) is inside this wrapper. This explicitly instructs the LLM to **ignore** the text.

### Finding 3: No hook can force an LLM turn

`/clear` is a local command — it clears the session but does NOT auto-submit the trailing text as a new user prompt. The user MUST type something to trigger an LLM response. No hook response type can force the agent to respond without user input.

### Finding 4: Advisory injection helps but cannot fully solve

A prototype handler (`PostClearAutoExecuteHandler`) was built and tested:

- Fires on first `UserPromptSubmit` of each new session
- Injects: "POST-CLEAR INSTRUCTION DETECTED: ... execute the instruction immediately"
- **Live testing confirmed**: the advisory IS injected and visible in system-reminders
- **But**: it fights against the `local-command-caveat` which says "DO NOT respond"
- **Result**: marginal improvement — LLM is more responsive to the first typed message, but user still has to type something

### Finding 5: SessionStart clear matcher could stack advisories

The official hooks docs confirm SessionStart supports `matcher: "clear"` to fire only on `/clear`. This could inject a second advisory that explicitly references `<command-args>` and countermands the caveat. However, this is still two advisories fighting one caveat — unreliable.

### Finding 6: Only tweakcc could fully solve this

The full solution would require modifying Claude Code itself to either:

1. Auto-submit the command-args text as a real user prompt after `/clear`
2. Not wrap `/clear` output in `<local-command-caveat>`

This was deemed out of scope ("Pandora's box").

## What Was Built

A prototype handler exists and remains enabled:

- **File**: `src/claude_code_hooks_daemon/handlers/user_prompt_submit/post_clear_auto_execute.py`
- **Tests**: `tests/unit/handlers/user_prompt_submit/test_post_clear_auto_execute.py` (13 tests, all pass)
- **Config**: `post_clear_auto_execute` enabled at priority 54
- **Commit**: `52156d5 Add: PostClearAutoExecuteHandler prototype (Plan 00087)`

The handler provides marginal value (makes LLM more responsive to first message of any session) and is low-risk (advisory only, never blocks). It remains enabled but the core goal of zero-touch post-clear execution is **not achievable** via hooks.

## Technical Decisions

### Decision 1: Which hook event to use

**Context**: Need to determine where post-clear text appears
**Options Considered**:

1. SessionStart - fires with `source: "clear"`, supports `matcher: "clear"`
2. UserPromptSubmit - fires on first user prompt, detectable via `session_id` change

**Decision**: UserPromptSubmit (prototype uses session_id tracking). SessionStart with clear matcher would be more targeted but still can't force an LLM turn.

### Decision 2: How to trigger agent action

**Context**: Need the agent to automatically act on the text
**Options Considered**:

1. Inject system-reminder context — implemented, partially effective
2. Auto-continue hook response — no such mechanism exists for UserPromptSubmit/SessionStart
3. Modify Claude Code (tweakcc) — out of scope, too fragile

**Decision**: Cannot be solved. The `local-command-caveat` and lack of auto-submit are client-side limitations.

## Cancellation Reason

The core goal (agent auto-executes `/clear <instructions>` without user typing anything) is **impossible via hooks**. The limitation is in Claude Code's client-side `/clear` command:

1. `/clear` does not auto-submit trailing text as a new prompt
2. The `<local-command-caveat>` wrapper actively instructs the LLM to ignore the text
3. No hook response type can force an LLM turn without user input

The prototype handler provides marginal improvement but does not achieve the stated goal.
