## Problem Statement

### The Dogfooding Failure

During plan 00080 implementation, the agent used dismissive language ("no issues with my new code") to deflect shellcheck warnings. The `DismissiveLanguageDetectorHandler` — a Stop event handler designed to catch exactly this — failed to fire.

Investigation revealed **three compounding failures**:

1. **Stop events rarely fire**: Claude Code's Stop hook fires on session termination, not after every turn. GitHub issue #29881 confirms this upstream limitation. In an entire multi-hour session, only 1 Stop event reached the daemon.

2. **TranscriptReader crash**: When the Stop event DID fire, both the hedging and dismissive handlers crashed with `TypeError: expected string or bytes-like object, got 'list'` — a bug where real Claude Code transcripts use `type=assistant` with content as a list of blocks, but the parser's legacy path stored the list directly. (Fixed in commit `226b43c`.)

3. **Pattern gaps**: Even if the handler had run, the phrases used ("no issues with my new code", "not introduced by my change") didn't match existing regex patterns. (Also fixed in `226b43c`.)

### The Fundamental Problem

**Quality enforcement handlers that depend on Stop events are unreliable by design.** The Stop event is the wrong inflection point for transcript auditing because:

- It fires too infrequently (once per session, not per turn)
- It fires too late (after the agent has already finished responding)
- It cannot block or correct behaviour (the response is already sent)

We need a way to audit assistant responses **proactively**, at a point where we can still intervene.

### The Opportunity

PreToolUse fires on **every tool call** — reliably, frequently, and at a point where the tool can be blocked. This makes it the ideal inflection point for transcript auditing. But "audit the transcript for dismissive language" is not really about the tool being used — it's a **different kind of check entirely**.

This suggests a new architectural concept: **pseudo-events**.

## Background Research

### Current Event Model

Claude Code fires these real events:

| Event | When | Frequency | Can Block? |
|-------|------|-----------|------------|
| SessionStart | Session begins | Once | No |
| UserPromptSubmit | User sends message | Per message | Yes |
| PreToolUse | Before tool execution | Per tool call (HIGH) | Yes |
| PostToolUse | After tool execution | Per tool call (HIGH) | No (advisory) |
| Stop | Agent finishes responding | Unreliable/rare | No (advisory) |
| SubagentStop | Subagent finishes | Per subagent | No (advisory) |
| Notification | Agent sends notification | Per notification | No |
| PreCompact | Before context compaction | Per compaction | No |
| SessionEnd | Session ends | Once | No |

**Key insight**: PreToolUse is the only high-frequency event that can block. All quality enforcement currently either:
- Uses PreToolUse (blocking handlers for dangerous commands) — **reliable**
- Uses Stop/PostToolUse (advisory handlers for language quality) — **unreliable**

### Transcript Message Identity

Every transcript entry has stable unique identifiers:

```
uuid:        "26fd317e-..."  — unique per entry
parentUuid:  "c165f69b-..."  — forms a linked chain
sessionId:   "747c4a57-..."  — same for entire session
timestamp:   ISO-8601         — ordering
```

This enables **incremental auditing**: track the last audited message UUID, only scan new messages since then.

### What "Nitpicking" Means

The term captures the intent: these are **quality micro-checks** that don't warrant blocking a tool call on their own merit, but collectively enforce a standard of communication. Examples:

- Dismissive language detection (deflecting responsibility)
- Hedging language detection (excessive uncertainty)
- Commit message quality (too vague, too long)
- Code comment quality (obvious comments, missing complex ones)
- Response verbosity (too wordy, not concise enough)
- Promise tracking (agent said "I'll do X" but hasn't)

None of these are about the tool being used. They're about **the quality of the agent's recent behaviour**.

## Concept: Pseudo-Events

### Definition

A **pseudo-event** is a synthetic event generated by the daemon itself, triggered by a real event but representing a different semantic concern. It has its own handler chain, its own routing, and its own configuration — but it piggybacks on a real event's execution lifecycle.

### How It Works

```
Real Event Flow:
  Claude Code → PreToolUse → FrontController → Handler Chain → Response

Pseudo-Event Flow:
  Claude Code → PreToolUse → FrontController → Handler Chain
                                                    ↓
                                              NitpickHandler (PreToolUse handler)
                                                    ↓
                                              Generates pseudo-event
                                                    ↓
                                              Nitpick Handler Chain
                                                    ↓
                                              Aggregated results
                                                    ↓
                                              Block/Advisory response
```

### Properties

1. **Triggered by real events**: A pseudo-event doesn't exist without a real event to trigger it
2. **Own handler chain**: Pseudo-event handlers are registered, configured, and dispatched separately
3. **Own configuration**: `handlers.nitpick.dismissive_language: {enabled: true}` etc.
4. **Composable results**: Multiple nitpick checkers can fire, results are aggregated
5. **Inherits blocking capability**: If the triggering event can block (PreToolUse), so can the pseudo-event

### Why Not Just More PreToolUse Handlers?

1. **Semantic clarity**: "Check for dismissive language" is not about the tool — mixing it into the PreToolUse chain conflates concerns
2. **Configuration namespacing**: Nitpick handlers should be grouped under `handlers.nitpick.*`, not `handlers.pre_tool_use.*`
3. **Shared infrastructure**: All nitpick handlers need transcript reading, incremental auditing, and result aggregation — this is pseudo-event infrastructure, not per-handler boilerplate
4. **Extensibility**: Adding a new nitpick check should be as simple as adding a strategy class, not wiring up a full PreToolUse handler with transcript reading logic

### Future Pseudo-Events

The pattern could extend beyond nitpick:

- **TurnComplete**: Synthetic event fired on PreToolUse when the previous turn's tool results have been processed (approximates the missing `TurnComplete` event the community wants)
- **ContextPressure**: Synthetic event when context usage exceeds thresholds
- **IdleCheck**: Synthetic event when no tool calls for N seconds

## Design Considerations

### Incremental Auditing

```python
class NitpickState:
    last_audited_uuid: str | None  # UUID of last audited message
    last_audited_line: int         # Line number (fast seek optimisation)
    findings_count: int            # Running count of findings this session
```

- Persisted in DaemonDataLayer (in-memory, lost on restart — acceptable)
- On each trigger: read transcript from last_audited_uuid forward
- Audit only new assistant messages
- Update last_audited_uuid after audit

### Configurable Audit Window

```yaml
handlers:
  nitpick:
    enabled: true
    max_messages: 5          # Max messages to look back (cap)
    trigger_frequency: 1     # Every Nth PreToolUse (1 = every call)
    mode: advisory           # advisory | blocking
    dismissive_language:
      enabled: true
    hedging_language:
      enabled: true
    # Future: commit_message_quality, response_verbosity, etc.
```

### Blocking vs Advisory

Two modes:
- **Advisory** (default): Inject context warning but allow tool to proceed
- **Blocking**: Block the tool call, forcing the agent to acknowledge and correct

Advisory is safer as default — blocking every tool call for a language quality issue could be disruptive.

### Relationship to Existing Stop Handlers

The existing Stop-based `DismissiveLanguageDetectorHandler` and `HedgingLanguageDetectorHandler` would remain as a **secondary safety net** but the primary enforcement shifts to the nitpick pseudo-event. Over time, if Stop events remain unreliable, the Stop handlers could be deprecated.

## Resolved Design Decisions

### Q1: Strategies (Protocol), not full Handlers

**Decision**: NitpickChecker as a Protocol, not Handler ABC.

**Rationale**: Checkers are lightweight pattern matchers, not event handlers. They don't need tags, priority, terminal flags, or acceptance tests — those belong to the NitpickHandler itself. A checker just needs:

```python
class NitpickChecker(Protocol):
    checker_id: str
    def check(self, text: str) -> list[NitpickFinding]: ...
```

Adding a new checker = one file with patterns. No wiring, no registration boilerplate.

### Q2: Internal dispatch, not first-class pseudo-events

**Decision**: NitpickHandler dispatches to checkers internally. No EventRouter changes.

**Rationale**: YAGNI. The NitpickHandler is a normal PreToolUse handler that internally runs checker strategies. The "pseudo-event" concept is documented as an architectural pattern for future reference, but building first-class infrastructure now would be premature. If we later need TurnComplete, ContextPressure, or IdleCheck pseudo-events, we extract the pattern then.

### Q3: Reset audit state on compaction

**Decision**: Reset `last_audited_uuid` and `last_byte_offset` to zero on context compaction.

**Rationale**: After compaction, the agent's context is reset. Re-auditing a few messages (~5 max from audit window) is harmless at <2ms. The alternative (preserving state across compaction via PreCompact) adds complexity for negligible benefit.

### Q4: Performance — benchmarked, no concerns

**Decision**: Use byte-offset incremental reading. Full parse is fast enough as fallback.

**Benchmarks** (3.9MB transcript, 1732 entries, 426 assistant messages):
- Full parse: **15.8ms** (well under 50ms budget)
- Seek-based (last 500KB): **1.8ms**
- Last 100 lines: **4.5ms**

**Incremental approach**: Store `last_byte_offset` in NitpickState. On each trigger, seek to offset, read only new lines, parse only new entries. Expected overhead: **<2ms per PreToolUse call**.

**Fallback**: If offset becomes stale (file truncated/rotated), fall back to full parse at 15.8ms — still safe.

### Q5: Trigger on every PreToolUse call (frequency=1)

**Decision**: Default `trigger_frequency: 1` (every call).

**Rationale**: With <2ms incremental overhead, there's no performance reason to skip calls. Every PreToolUse = every tool call = every turn boundary. Missing a call means potentially missing dismissive language until the next call.

The `trigger_frequency` config option is kept for users who want to reduce overhead in very long sessions, but defaults to every call.
